1 ปลายทาง “ในวง” หรือ “ข้ามวง”?

ก่อนส่งอะไรออกไป PC ต้องตอบคำถามแรกให้ได้ก่อน: ปลายทางอยู่วงเดียวกับฉันไหม? วิธีตอบคือเอา IP ตัวเองกับปลายทางมา AND กับ subnet mask แล้วเทียบว่าได้ network เดียวกันหรือเปล่า

ลองเล่น: เครื่องคิดเลข “ในวง/ข้ามวง”

กดเลือกสถานการณ์ แล้วดูว่า PC ตัดสินใจอย่างไร (และทำไม mask ผิดถึงพัง)

⚠️
ข้อควรระวัง Subnet mask ผิดแม้นิดเดียว ทำให้ PC ตัดสินใจ “ในวง/ข้ามวง” ผิดทันที บางเคส ping ในวงได้แต่ข้ามวงไม่ได้ หรือกลับกัน — เป็นกับดักยอดฮิตในข้อสอบและงานจริง
🌍
กรณีใช้งานจริง ผู้ใช้แจ้งว่า “เข้าอินเทอร์เน็ตได้ แต่เข้าเซิร์ฟเวอร์ในบริษัทไม่ได้” หลายครั้งต้นเหตุคือ mask ที่เครื่องตั้งผิด ทำให้เครื่องคิดว่าเซิร์ฟเวอร์อยู่คนละวงแล้วยิงไป Gateway ที่ไม่มี route ไปหาเซิร์ฟเวอร์นั้น

2 ARP: แปลง IP ให้เป็น MAC

Ethernet ส่งข้อมูลด้วย MAC เท่านั้น เมื่อ PC-A รู้แค่ IP ของ Gateway แต่ไม่รู้ MAC มันจะ broadcast ถามทั้งวงว่า “ใครคือ 192.168.10.1?” นี่คือ ARP (Address Resolution Protocol)

💡
ตัวอย่าง บนเครื่องจริงลองพิมพ์ arp -a (Windows) หรือ ip neigh (Linux) จะเห็นตาราง IP→MAC ที่เครื่องจำไว้ ถ้า entry ของ Gateway หายหรือขึ้นเป็น incomplete มักแปลว่า ARP ไม่สำเร็จ
🛑
ข้อควรระวัง ARP ไม่มีการยืนยันตัวตน จึงถูกปลอมได้ (ARP spoofing) ในงาน Security นี่คือช่องโหว่คลาสสิก ที่ทำ Man-in-the-Middle — ของจริงจึงใช้ DAI (Dynamic ARP Inspection) และ DHCP Snooping ป้องกัน
🌍
กรณีใช้งานจริง IP ชนกัน (duplicate IP) จะเห็นอาการ ping ติด ๆ ดับ ๆ เพราะ ARP reply มาจากสองเครื่องสลับกัน — ไล่จับได้จากการดู MAC ใน ARP table ที่เปลี่ยนไปมา

3 ประกอบ Frame แล้วเข้าใจ “อะไรเปลี่ยน/อะไรอยู่”

เมื่อได้ MAC ของ hop ถัดไปแล้ว PC-A ห่อข้อมูลเป็น frame โครงสร้างคร่าว ๆ คือ Ethernet ห่อ IP ห่อ ICMP อีกที

โครงของ frame ที่ออกจาก PC-A
┌─────────────── Ethernet (Layer 2) ───────────────┐
│ dst MAC = c1:c1:01 (Gateway R1)   src MAC = aa:aa:0a │
│   ┌───────────── IP (Layer 3) ─────────────┐        │
│   │ dst IP = 192.168.20.20 (PC-B)  src IP = .10.10 │
│   │   ┌──── ICMP ────┐                      │        │
│   │   │ Echo Request │  TTL = 128           │        │
│   │   └──────────────┘                      │        │
│   └─────────────────────────────────────────┘        │
└───────────────────────────────────────────────────────┘

จุดที่ต้องเก็บให้ขึ้นใจ: ในขณะเดียวกัน MAC ปลายทางชี้ที่ Gateway แต่ IP ปลายทางชี้ที่ PC-B นี่คือเหตุผลที่ networking ต้องมีสองที่อยู่ ไม่ใช่อย่างใดอย่างหนึ่ง

frame เดิม เดินทาง 3 ช่วง — ดูว่าหัวเปลี่ยนตรงไหน

ช่วงเส้นทางsrc MACdst MACsrc IPdst IPTTL
PC-A → R1 aa:aa:0a c1:c1:01 .10.10 .20.20 128
R1 → PC-B c1:c1:01 bb:bb:14 .10.10 .20.20 127
🧠
ทำไมถึงเป็นแบบนั้น คอลัมน์ MAC เปลี่ยนทั้งคู่ เมื่อผ่าน R1 เพราะเข้าสู่วงใหม่ (VLAN 20) ต้องใช้ MAC ของวงนั้น ส่วน IP ไม่ขยับ เพราะมันคือปลายทางจริง และ TTL ลด 1 ทุกครั้งที่ Router forward เพื่อกัน loop
🌍
กรณีใช้งานจริง คำสั่ง traceroute อาศัยหลัก TTL นี้ตรง ๆ — มันส่ง packet ที่ TTL น้อย ๆ เพื่อให้ router แต่ละ hop ตอบกลับมา ทำให้เราเห็นเส้นทางทีละ hop ว่า packet ไปตายที่ไหน

4 Switch + VLAN + Trunk (802.1Q)

Switch ไม่สนใจ IP เลย มันดูแค่ MAC ปลายทาง แล้วเปิด MAC Address Table เพื่อเลือกพอร์ตส่งออก

SW1# show mac address-table (ตัวอย่าง)
VLAN    MAC Address       Type      Ports
10      aa:aa:0a          DYNAMIC   Fa0/1     ← PC-A
20      bb:bb:14          DYNAMIC   Fa0/2     ← PC-B
10      c1:c1:01          DYNAMIC   Gi0/1     ← R1 ผ่าน trunk
20      c1:c1:01          DYNAMIC   Gi0/1     ← R1 ผ่าน trunk

พอปลายทางเป็น R1 ซึ่งอยู่หลัง trunk (สายเส้นเดียวที่แบกหลาย VLAN) SW1 จะ ติดแท็ก 802.1Q ระบุหมายเลข VLAN ลงไปในหัว frame ก่อนส่งออก เพื่อให้ปลายทางแยกออกว่า frame นี้เป็นของ VLAN ไหน

ชนิดพอร์ตทำอะไรกับ VLAN tagต่อกับ
Accessไม่มี tag (frame ล้วน) เป็นสมาชิก VLAN เดียวเครื่องผู้ใช้ / PC
Trunkใส่/อ่าน tag 802.1Q แบกได้หลาย VLANSwitch↔Switch หรือ Switch↔Router
⚠️
ข้อควรระวัง: Native VLAN mismatch สองฝั่งของ trunk ต้องตั้ง native VLAN ตรงกัน ถ้าไม่ตรง VLAN ที่วิ่งแบบ untagged จะรั่วข้ามกัน (VLAN hopping) เป็นทั้งบั๊กและช่องโหว่ความปลอดภัย
✅
Best Practice ตั้ง trunk แบบระบุ VLAN ที่อนุญาตด้วย switchport trunk allowed vlan และเปลี่ยน native VLAN ให้เป็น VLAN ที่ไม่ได้ใช้งานจริง อย่าปล่อยเป็น VLAN 1 ตามค่าเริ่มต้น

5 Routing + Longest Prefix Match

R1 แกะ tag ออก อ่าน IP ปลายทาง แล้วเปิด Routing Table ปัญหาคือบ่อยครั้ง มีหลาย route ที่ “ครอบคลุม” ปลายทางเดียวกัน Router จึงใช้กฎ Longest Prefix Match: เลือกเส้นที่ prefix ยาวที่สุด (เจาะจงที่สุด) เสมอ

ลองเล่น: เลือกปลายทาง แล้วดูว่า R1 เลือก route ไหน

ปลายทาง:
Route (network/prefix)Prefixออกทางเทียบกับปลายทาง
192.168.20.32/28/28via 10.0.0.9
192.168.20.0/24/24connected G0/0.20
192.168.0.0/16/16via 10.0.0.2
0.0.0.0/0/0via 203.0.113.1
🧠
ทำไมต้อง “ยาวสุด” prefix ยิ่งยาว = ยิ่งเจาะจงกลุ่ม IP เล็กลง Router เลือกอันเจาะจงสุดเพื่อให้ผลลัพธ์ เดียว ไม่กำกวม เหมือนที่อยู่ “บ้านเลขที่ 20 ซอย 3” ย่อมเจาะจงกว่า “แขวงนี้ทั้งแขวง”
🌍
กรณีใช้งานจริง Default route 0.0.0.0/0 คือ “ถ้าไม่ตรงอะไรเลย ให้ส่งออกทางนี้” — เป็นเส้นที่สั้นที่สุด จึงถูกใช้เป็นทางเลือกสุดท้ายเสมอ นี่คือหลักการเดียวกับที่ router บ้านคุณส่งทุกอย่างที่ไม่ใช่วงในออกไปหา ISP

6 ขากลับ — และทำไม “ping ไม่กลับ” ถึงเจอบ่อย

ถึงมือ PC-B แล้วยังไม่จบ — PC-B ต้องส่ง Echo Reply กลับ โดยสลับ src/dst IP แล้ววิ่งย้อนกระบวนการเดิมทั้งหมด (ตัดสินใจในวง/ข้ามวง → ARP → frame → switch → routing) กว่า ping จะขึ้นว่า “สำเร็จ” ต้องครบ ทั้งไปและกลับ

🛑
กับดักที่พบบ่อยที่สุด เวลาเจอ “ping ไม่ได้” มือใหม่มักไล่แต่ขาไป แต่จริง ๆ ปัญหาส่วนใหญ่คือ ขากลับไม่มี route — ขาไปถึงปลายทางเรียบร้อย แต่ปลายทางไม่รู้ทางกลับมาหาเรา ต้องมี route ครบ ทั้งสองทิศเสมอ

ลำดับการไล่ปัญหาแบบ Engineer (follow-the-path)

  1. L1
    ชั้นกายภาพ/ลิงก์ — สายต่อไหม พอร์ต up ไหม (show ip interface brief) VLAN ถูกไหม
  2. L2
    ในวงเดียวกัน — ARP ขึ้นไหม MAC เข้า table ไหม trunk/native VLAN ตรงไหม
  3. L3
    ข้ามวง — routing table มีเส้นทางไป และ กลับไหม gateway/next-hop ถูกไหม mask ถูกไหม
  4. ↔
    ยืนยันสองทิศ — ping จากทั้งสองฝั่ง และ traceroute ดูว่าตายที่ hop ไหน
✅
Best Practice ไล่จากล่างขึ้นบน (L1 → L2 → L3) และ แบ่งครึ่งเส้นทาง (divide & conquer) — ping จากจุดกึ่งกลางก่อน เพื่อฟันธงว่าปัญหาอยู่ครึ่งไหน จะเร็วกว่าเดาสุ่มมาก

? คำถามที่พบบ่อย (FAQ)

ทำไมต้องมีทั้ง MAC และ IP ใช้อย่างเดียวไม่ได้เหรอ?

IP ออกแบบให้ “มีลำดับชั้น” (network + host) จึงสรุปเส้นทางและ route ข้ามโลกได้ ส่วน MAC เป็นเลขประจำการ์ดที่แบนราบ ไม่มีโครงสร้างเชิงภูมิศาสตร์ เหมาะกับการส่งภายในวงเดียวเท่านั้น สองอย่างทำคนละหน้าที่จึงต้องมีคู่กัน

Router-on-a-Stick กับ Layer 3 Switch (SVI) ต่างกันยังไง?

ROAS ใช้ Router พอร์ตเดียวแบ่งเป็น subinterface ต่อ VLAN ผ่าน trunk — ประหยัดพอร์ตแต่ traffic ข้าม VLAN ทั้งหมดวิ่งผ่านสายเส้นเดียว (คอขวด) ส่วน L3 Switch ทำ routing ในตัวด้วย SVI ความเร็วระดับฮาร์ดแวร์ เหมาะกับเครือข่ายที่โตขึ้น ดูรายละเอียดใน Lab 4 และ Lab 5

ARP cache หมดอายุแล้วเกิดอะไรขึ้น?

เมื่อ entry หมดอายุ (ปกติไม่กี่นาที) การส่งครั้งถัดไปจะทริกเกอร์ ARP รอบใหม่ก่อน 1 ครั้ง ทำให้ packet แรกดีเลย์นิดหน่อยหรือหลุดไป 1 ตัว แล้วจึงกลับมาปกติ — ไม่ใช่ปัญหา แต่ช่วยอธิบายว่าทำไม ping ตัวแรกบางทีช้ากว่าตัวถัด ๆ ไป

TTL คืออะไร ทำไมต้องลดลงทุก hop?

TTL (Time To Live) คือตัวนับกัน packet วนไม่รู้จบเวลามี routing loop ทุกครั้งที่ Router forward จะลด TTL ลง 1 ถ้าถึง 0 จะถูกทิ้งแล้วส่ง ICMP Time Exceeded กลับ — traceroute ใช้กลไกนี้แหละในการวาดเส้นทาง

ทำไมเลขปลายทางเดียวกันถึง match ได้หลาย route?

เพราะ route เขียนเป็นช่วง (network/prefix) ที่ซ้อนกันได้ เช่น 192.168.20.20 อยู่ทั้งใน /24, /16 และ /0 กฎ Longest Prefix Match จึงมีไว้ตัดสินให้เหลือคำตอบเดียว ลองกดเล่นใน widget ของ Step 5 ได้
