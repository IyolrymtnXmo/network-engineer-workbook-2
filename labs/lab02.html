Static + Default Route และ Longest Prefix Match

Lab นี้พาคุณออกแบบเส้นทางแบบ static ข้ามเครือข่าย 3 router (สาขา – สำนักงานใหญ่ – ISP) ฝึกอ่าน routing table ด้วยหลัก Longest Prefix Match และเจอปัญหาคลาสสิกของงานจริง: "ขาไปถึง แต่ขากลับไม่มี" ซึ่งเป็นสาเหตุอันดับต้น ๆ ของเคส ping ไม่ได้ที่ SI ทุกบริษัทต้องเจอ

ระดับ: พื้นฐาน–กลาง ⏱ ใช้เวลาประมาณ 75 นาที Default Route Longest Prefix Match Routing Table

1 Learning Objectives

หลังจบ Lab นี้ ผู้เรียนต้อง "ทำได้" ตามรายการต่อไปนี้ ไม่ใช่แค่ "เคยอ่านผ่าน" — reviewer จะทดสอบด้วยการให้อธิบาย routing table จริงทีละบรรทัด

  • ตั้งค่า static route แบบเจาะจง (specific route) และ default route (ip route 0.0.0.0 0.0.0.0 ...) บน Cisco IOS ได้ถูกต้องตาม design
  • อธิบายได้ว่าเมื่อ packet หนึ่งใบมาถึง router เลือกเส้นทางอย่างไรด้วยหลัก Longest Prefix Match (LPM) พร้อมยกตัวอย่างจาก routing table จริง
  • แยกความหมายของค่า Administrative Distance (AD) กับ metric และบอกได้ว่าแต่ละค่าใช้ตัดสินตอนไหน
  • อ่าน output ของ show ip route ได้ครบ: code (C / L / S / S*), ค่า [AD/metric], next-hop, gateway of last resort
  • ออกแบบ routing ให้ stub network (สาขาเล็กที่มีทางออกทางเดียว) ด้วย default route เส้นเดียว และอธิบายได้ว่าทำไมไม่ควรใส่ route ละเอียดโดยไม่จำเป็น
  • วิเคราะห์ปัญหา "ping จาก router ได้ แต่ ping จากเครื่องลูกข่ายไม่ได้" และชี้ได้ว่าเกิดจาก return route ที่ปลายทางไม่มี
  • ตรวจจับ configuration ที่ผิดจากเอกสารที่ได้รับมา (addressing table / initial config) ก่อนนำไปใช้งานจริง

4 Lab Guide — สถานการณ์

ลูกค้าของบริษัทคุณเป็นธุรกิจซื้อ-ขายวัสดุก่อสร้าง มีสำนักงานใหญ่ (HQ) หนึ่งแห่งและสาขา (Branch) หนึ่งแห่ง เชื่อมกันด้วยลิงก์ WAN แบบ point-to-point ที่เช่าจากผู้ให้บริการ ทาง HQ ต่อออก Internet ผ่าน router ของ ISP (ใน Lab นี้คือ R3 ซึ่งมี Loopback0 = 8.8.8.8/32 จำลองปลายทางบน Internet) ทีม engineer ชุดก่อนได้ตั้งค่า R1 ที่สาขาไว้บางส่วนแล้วก่อนส่งมอบงานต่อให้คุณ

Design Intent (แบบที่ได้รับอนุมัติแล้ว)

  • R1 (Branch) — เป็น stub network มีทางออกทางเดียว จึงกำหนดให้ใช้ default route เส้นเดียว ชี้ next-hop ไปที่ 10.0.12.2 ห้ามใส่ route เจาะจงรายเครือข่าย เพื่อให้ config สาขาเรียบง่ายและไม่ต้องตามแก้ทุกครั้งที่ HQ เพิ่ม subnet ใหม่
  • R2 (HQ) — ต้องรู้จัก Branch LAN แบบเจาะจง: static route 192.168.10.0/24 ชี้ไป 10.0.12.1 และมี default route ชี้ออก ISP ที่ 10.0.23.2 สำหรับปลายทางอื่นทั้งหมด
  • R3 (ISP) — อยู่นอกขอบเขตการดูแลของเรา ทีม ISP จะเพิ่ม static route ให้เฉพาะรายการที่ลูกค้าร้องขอเป็นลายลักษณ์อักษรเท่านั้น ใบร้องขอ (route request form) ฉบับปัจจุบันที่ทีมก่อนหน้าส่งไว้มีรายการเดียว: เครือข่าย 10.0.12.0/30 ชี้ next-hop 10.0.23.1

เกณฑ์ตรวจรับของลูกค้า (acceptance test): PC1 ↔ PC2 ต้อง ping ถึงกัน และทั้ง PC1 กับ PC2 ต้อง ping 8.8.8.8 ได้

5 Configuration Tasks

ทำตามโจทย์ทีละข้อ ยังไม่ต้องเปิดเฉลย — ติ๊กถูกเมื่อทำเสร็จ ระบบจะจำไว้ให้

  1. สร้าง topology และนำ Initial Configuration ไปใช้กับ R1

    สร้างอุปกรณ์ตามผัง Section 2 (router 3 ตัว + PC 2 เครื่อง) เชื่อมสายตามตารางการเชื่อมต่อ จากนั้นนำ Initial Configuration ของ R1 ใน Section 4 ไปใช้ทั้งชุด — ถือว่า config ชุดนี้คือสภาพที่ทีมก่อนหน้าส่งมอบไว้

  2. ตั้งค่าพื้นฐานและ IP address บน R2

    ตั้ง hostname เป็น R2 แล้วกำหนด IP address ให้ G0/0, G0/1, G0/2 ตาม Addressing Table พร้อมใส่ description ทุก interface และเปิดใช้งานให้สถานะเป็น up/up

  3. ตั้งค่าพื้นฐานและ IP address บน R3

    ตั้ง hostname เป็น R3 กำหนด IP address ให้ G0/1 และสร้าง Loopback0 เป็น 8.8.8.8/32 เพื่อจำลองปลายทางบน Internet ตรวจสอบข้อมูลที่ใช้กับเอกสารประกอบให้รอบคอบก่อนพิมพ์

  4. ตั้งค่า IP ให้ PC1 / PC2 และทดสอบภายใน LAN

    กำหนด IP address, subnet mask และ default gateway ให้ PC ทั้งสองเครื่องตาม Addressing Table แล้วทดสอบ: PC1 ต้อง ping 192.168.10.1 ได้ และ PC2 ต้อง ping 192.168.20.1 ได้ ก่อนไปข้อถัดไป

  5. ทำให้ routing ของ R1 เป็นไปตาม design intent

    ตาม design R1 ต้องมี default route เส้นเดียวชี้ next-hop 10.0.12.2 และห้ามมี static route อื่น ทดสอบว่า routing table ของ R1 มี gateway of last resort ถูกต้อง และ R1 ping 10.0.12.2 ได้

  6. ตั้งค่า static + default route บน R2

    เพิ่ม route เจาะจงไปยัง Branch LAN 192.168.10.0/24 ชี้ next-hop 10.0.12.1 และ default route ชี้ next-hop 10.0.23.2 จากนั้นทดสอบ: PC1 ↔ PC2 ต้อง ping ถึงกันได้แล้วในขั้นนี้

  7. ตั้งค่า route บน R3 ตามใบร้องขอของลูกค้า

    สวมบทบาทเป็นทีม ISP: เพิ่ม static route ไปยังเครือข่าย 10.0.12.0/30 ชี้ next-hop 10.0.23.1 ตามใบร้องขอฉบับปัจจุบัน

  8. ทดสอบตามเกณฑ์ตรวจรับ และวิเคราะห์ผล

    ทดสอบครบ 4 รายการ: (ก) PC1 ping PC2, (ข) PC1 ping 8.8.8.8, (ค) PC2 ping 8.8.8.8, (ง) R1 ping 8.8.8.8 — จดผลแต่ละรายการ ถ้ารายการใดไม่ผ่าน ให้ไล่ routing table ของ router ทุกตัวบนเส้นทางทั้งขาไปและขากลับ หาสาเหตุและแก้ให้ผ่านเกณฑ์ตรวจรับให้ได้ด้วยตัวเองก่อนเปิดเฉลย

6 Verification

ตรวจสอบอย่างเป็นระบบ ไม่ใช่แค่ ping แล้วจบ — engineer ต้องพิสูจน์ได้ว่า "ทำไมมันใช้ได้" ด้วยหลักฐานจากคำสั่ง show เสมอ

show ip route

ดู routing table ทั้งหมดของ router — คำสั่งที่สำคัญที่สุดของ Lab นี้

  • code C = connected, L = local (/32 ของ interface ตัวเอง), S = static, S* = static ที่เป็น candidate default
  • ค่าในวงเล็บ [1/0] คือ [AD/metric] — static route ต้องเป็น AD 1, metric 0
  • บรรทัด Gateway of last resort is ... ต้องชี้ next-hop ที่ถูกต้อง ถ้าขึ้น "is not set" แปลว่ายังไม่มี default route ที่ใช้งานได้
show ip route 8.8.8.8

ถาม router ตรง ๆ ว่า "ปลายทางนี้ ใช้ route ไหน" — ใช้ตรวจผลของ Longest Prefix Match กับปลายทางเจาะจง

  • ถ้ามี route เจาะจง match จะแสดง "Routing entry for ..." พร้อม next-hop, AD และ metric
  • ข้อควรรู้: ถ้าปลายทาง match ได้แค่ default route คำสั่งนี้จะตอบ % Network not in table ทั้งที่ packet ยังถูก forward ผ่าน S* ได้ตามปกติ — อย่าสรุปผิดว่า "route หาย"
show ip interface brief

ตรวจสถานะ interface และ IP address ทุกขาในหน้าจอเดียว

  • ทุก interface ที่ใช้งานต้องเป็น up / up — ถ้า Status = administratively down แปลว่าลืม no shutdown
  • เทียบ IP address ที่ขึ้นจริงกับ design ทีละบรรทัด อย่าเชื่อเอกสารอย่างเดียว
ping 8.8.8.8 source GigabitEthernet0/0

ping จาก router โดยบังคับ source เป็นขา LAN — ใช้จำลองมุมมองของเครื่องลูกข่ายโดยไม่ต้องเดินไปที่ PC

  • ping ปกติจาก router จะใช้ source เป็น IP ของ interface ขาออก (ขา WAN) ซึ่งปลายทางมักรู้จัก — ผลทดสอบจึง "สวยเกินจริง"
  • ถ้า ping ธรรมดาผ่านแต่ใส่ source ขา LAN แล้วไม่ผ่าน ให้สงสัยทันทีว่าปลายทางไม่มี return route กลับมาหา subnet ของ LAN
traceroute 8.8.8.8

ไล่เส้นทางทีละ hop เพื่อดูว่า packet ตายที่ตรงไหน

  • hop สุดท้ายที่ตอบกลับมา คือ router ตัวสุดท้ายที่ยังส่งต่อ/ตอบได้ — จุดที่ต้องไปตรวจคือ hop ถัดไปจากนั้น
  • บน PC (Windows) ใช้ tracert 8.8.8.8 ให้ผลแบบเดียวกัน
show running-config | include ip route

ดูว่าเราพิมพ์ static route อะไรลงไปแล้วบ้าง (มุมมอง config)

  • เทียบกับ show ip route เสมอ — บรรทัดที่อยู่ใน running-config อาจไม่ถูก install ลง routing table ถ้า next-hop resolve ไม่ได้
  • เวลาแก้ static route ต้อง no ip route ... บรรทัดเดิมออกก่อน ไม่อย่างนั้นจะมี route เก่าค้างซ้อนกัน

กติกาการเลือกเส้นทาง — ลำดับที่ต้องท่องให้ขึ้นใจ

1) Longest Prefix Match ชนะเสมอ — เปรียบเทียบ "ความยาว prefix" ของทุก route ที่ match ปลายทาง route ที่ prefix ยาวที่สุด (เจาะจงที่สุด) ถูกใช้ก่อนโดยไม่สน AD หรือ metric ใด ๆ

2) AD ใช้เมื่อ prefix เท่ากันแต่มาจากคนละ source — เช่น 10.1.1.0/24 มีทั้งจาก static (AD 1) และ OSPF (AD 110) router จะ install ตัวที่ AD ต่ำกว่าลง routing table

3) metric ใช้เปรียบเทียบภายใน protocol เดียวกัน — เช่น OSPF มี 2 เส้นทางไป prefix เดียวกัน จะเลือกเส้นที่ cost ต่ำกว่า

สถานการณ์เหล่านี้เจอจริงหน้างานทั้งหมด ฝึก "วิธีคิด" ให้จบก่อน แล้วค่อยเทียบกับเฉลยใน Section 11

⚠ Scenario 7.1 — R1 ping 8.8.8.8 ได้ แต่ PC1 ping ไม่ได้
อาการ
Engineer ทดสอบจาก R1 ด้วย ping 8.8.8.8 ได้ !!!!! สวยงาม จึงรายงานลูกค้าว่าระบบพร้อมใช้ แต่พอลูกค้าลองจาก PC1 กลับได้ Request timed out. ทุกครั้ง ทั้งที่ PC1 ping gateway 192.168.10.1 และ ping PC2 ได้ปกติ
สิ่งที่ผู้ใช้พบ
บน R1: Success rate is 100 percent (5/5) — บน PC1: Request timed out. ครบทั้ง 4 packet และ tracert 8.8.8.8 วิ่งผ่าน 192.168.10.1 → 10.0.12.2 แล้วหลังจากนั้นเป็น * * * ตลอด
วิธีคิด
คำถามสำคัญที่สุด: ping สองครั้งนี้ "source IP" เหมือนกันหรือไม่? ping จาก R1 โดย default ใช้ source เป็น IP ของ interface ขาออก (ขา WAN 10.0.12.1) แต่ ping จาก PC1 ใช้ source 192.168.10.10 — ปลายทางจะตอบกลับได้ก็ต่อเมื่อมี route กลับมาหา source นั้น ดังนั้นถ้าผลต่างกันตาม source ให้สงสัย return route ที่ฝั่งปลายทางทันที ไม่ใช่ฝั่งเรา
ขั้นตอนการตรวจสอบ
  1. ยืนยันอาการบน R1: ping 8.8.8.8 (ผ่าน) แล้วตามด้วย ping 8.8.8.8 source GigabitEthernet0/0 — ถ้าครั้งหลังไม่ผ่าน แสดงว่าปัญหาผูกกับ source 192.168.10.x จริง
  2. ไล่ขาไป: R1 มี default route → R2 มี default route → packet ถึง R3 ได้ ขาไปจึงไม่ใช่ปัญหา
  3. ไล่ขากลับ: ไปดูที่ R3 ด้วย show ip route 192.168.10.10 ว่ามี route กลับหา subnet ต้นทางหรือไม่
  4. เทียบกับรายการ route ที่ ISP config ให้จริง (show running-config | include ip route บน R3) กับสิ่งที่ระบบต้องการ
ลำดับการ Troubleshoot
เทียบ source IP ของการทดสอบทั้งสองแบบ → พิสูจน์ด้วย ping แบบระบุ source → ไล่ routing table ขาไปจนถึงปลายทาง → ไล่ routing table ขากลับ จากปลายทางมาหา source
⚠ Scenario 7.2 — ใส่ default route แล้ว แต่ gateway of last resort "is not set"
อาการ
Intern คนหนึ่ง config R1 ใหม่ตั้งแต่ต้น แล้วพิมพ์ ip route 0.0.0.0 0.0.0.0 10.0.23.2 (ตั้งใจชี้ "ออก Internet" เลยจำ IP ฝั่ง ISP มาใส่) คำสั่งกด Enter ผ่าน ไม่มี error ใด ๆ แต่ R1 ping ออกไปไหนไม่ได้เลยนอกจาก 10.0.12.2
สิ่งที่ผู้ใช้พบ
show running-config | include ip route เห็นบรรทัด default route อยู่ครบ แต่ show ip route ขึ้น Gateway of last resort is not set และไม่มีบรรทัด S* ปรากฏใน routing table
วิธีคิด
บรรทัดที่อยู่ใน running-config ไม่ได้แปลว่าถูกใช้งานเสมอไป — static route จะถูก install ลง routing table (RIB) ก็ต่อเมื่อ router สามารถ resolve next-hop ได้ (recursive lookup: ต้องมี route อื่นที่พาไปถึง next-hop นั้น) คำถามคือ R1 รู้จักทางไป 10.0.23.2 หรือไม่?
ขั้นตอนการตรวจสอบ
  1. เทียบสองมุมมอง: show running-config | include ip route กับ show ip route — ถ้า config มีแต่ RIB ไม่มี ให้สงสัย next-hop resolve ไม่ได้
  2. ตรวจว่า next-hop อยู่ใน subnet ที่ R1 ต่อตรงหรือไม่: show ip route 10.0.23.2 → ควรได้ % Network not in table ซึ่งยืนยันปัญหา
  3. ทบทวน topology: subnet ที่ R1 ต่อตรงคือ 10.0.12.0/30 — next-hop ที่ถูกต้องของ R1 ต้องเป็นเพื่อนบ้านที่อยู่บนลิงก์เดียวกันเท่านั้น
ลำดับการ Troubleshoot
เทียบ running-config กับ RIB → ตรวจการ resolve next-hop → แก้ next-hop ให้เป็นเพื่อนบ้านบนลิงก์ตรง
⚠ Scenario 7.3 — พิมพ์ ip route แล้วเจอ "% Inconsistent address and mask"
อาการ
Intern อีกคนเพิ่งเรียนเรื่อง ACL กับ OSPF มา พอจะเพิ่ม route ไป Branch LAN บน R2 จึงพิมพ์ ip route 192.168.10.0 0.0.0.255 10.0.12.1 แล้ว IOS ตอบกลับด้วยข้อความ error ทันที
สิ่งที่ผู้ใช้พบ
ข้อความ % Inconsistent address and mask และ route ไม่ถูกสร้าง
วิธีคิด
ค่า 0.0.0.255 คือ wildcard mask ซึ่งใช้กับ ACL และคำสั่ง network ของ OSPF/EIGRP เท่านั้น ส่วนคำสั่ง ip route ต้องใช้ subnet mask เสมอ (บิต 1 ต่อเนื่องจากซ้าย เช่น 255.255.255.0) IOS ตรวจพบว่า address กับ mask ไม่สอดคล้องกันจึงปฏิเสธ — จุดที่อันตรายกว่าคือ wildcard บางค่า "บังเอิญ" เป็น mask ที่ valid แล้วคำสั่งผ่านโดยความหมายเพี้ยนไปเลย
ขั้นตอนการตรวจสอบ
  1. อ่าน error แล้วแยกให้ออกว่า parameter ตัวไหนของคำสั่งคือปัญหา (ตำแหน่งที่สองของ ip route คือ subnet mask)
  2. ทบทวนว่าคำสั่งไหนใช้ subnet mask คำสั่งไหนใช้ wildcard mask แล้วแปลงค่าให้ถูก: /24 → 255.255.255.0
  3. หลังแก้แล้วตรวจผลด้วย show ip route 192.168.10.0 ว่า route ขึ้นเป็น S พร้อม [1/0]
ลำดับการ Troubleshoot
อ่าน error message ให้จบ → ระบุ parameter ที่ผิด → แก้ด้วยรูปแบบที่ถูกของคำสั่งนั้น → verify ด้วย show

8 Packet Flow

วิเคราะห์เส้นทางของ PC1 ping 8.8.8.8 ทีละ hop ทั้งขาไปและขากลับ (สมมุติว่าระบบ config สมบูรณ์แล้ว) — MAC address ที่ใช้: PC1 = 0050.7966.6801, R1 G0/0 = 0c1a.2b00.1000, R1 G0/1 = 0c1a.2b00.1001, R2 G0/1 = 0c2a.2b00.2001, R2 G0/2 = 0c2a.2b00.2002, R3 G0/1 = 0c3a.2b00.3001

  1. PC1 — ตัดสินใจส่งเข้า gateway

    PC1 เทียบปลายทาง 8.8.8.8 กับ subnet ตัวเอง (192.168.10.0/24) → อยู่คนละ subnet จึงต้องส่งผ่าน default gateway 192.168.10.1 — PC1 ทำ ARP หา MAC ของ gateway (ไม่ใช่ของ 8.8.8.8) แล้วสร้าง ICMP Echo Request โดย TTL เริ่มต้น 128

    SRC MAC: 0050.7966.6801 (PC1) | DST MAC: 0c1a.2b00.1000 (R1 G0/0) | SRC IP: 192.168.10.10 | DST IP: 8.8.8.8
  2. R1 — Longest Prefix Match ครั้งที่ 1

    R1 เปิด routing table หา route ที่ match 8.8.8.8 → มีเพียง S* 0.0.0.0/0 [1/0] via 10.0.12.2 เท่านั้นที่ match (connected ทั้งสองเส้นไม่ครอบ 8.8.8.8) → forward ออก G0/1 ไป next-hop 10.0.12.2, ลด TTL เหลือ 127, ถอด L2 header เดิมทิ้งแล้วสร้าง frame ใหม่ — IP ต้นทาง/ปลายทางไม่เปลี่ยน

    SRC MAC: 0c1a.2b00.1001 (R1 G0/1) | DST MAC: 0c2a.2b00.2001 (R2 G0/1) | SRC IP: 192.168.10.10 | DST IP: 8.8.8.8
  3. R2 — Longest Prefix Match ครั้งที่ 2

    R2 หา route ที่ match 8.8.8.8 → route เจาะจง (192.168.10.0/24, connected /30 ทั้งหลาย) ไม่มีตัวไหนครอบ 8.8.8.8 เหลือเพียง S* 0.0.0.0/0 via 10.0.23.2 → forward ออก G0/2, TTL เหลือ 126

    SRC MAC: 0c2a.2b00.2002 (R2 G0/2) | DST MAC: 0c3a.2b00.3001 (R3 G0/1) | SRC IP: 192.168.10.10 | DST IP: 8.8.8.8
  4. R3 — ปลายทางรับ packet และสร้างคำตอบ

    8.8.8.8 คือ Loopback0 ของ R3 เอง → R3 ไม่ forward ต่อ แต่ประมวลผลแล้วสร้าง ICMP Echo Reply โดยสลับ SRC/DST: ปลายทางใหม่คือ 192.168.10.10 — จุดชี้เป็นชี้ตายของขากลับ: R3 ต้องมี route ที่ match 192.168.10.10 ซึ่งก็คือ S 192.168.0.0/16 [1/0] via 10.0.23.1 (summary route ที่ครอบทั้ง Branch และ HQ LAN) ถ้าไม่มีบรรทัดนี้ Echo Reply จะถูก drop ทิ้งตรงนี้ทันที

    SRC MAC: 0c3a.2b00.3001 (R3 G0/1) | DST MAC: 0c2a.2b00.2002 (R2 G0/2) | SRC IP: 8.8.8.8 | DST IP: 192.168.10.10
  5. R2 — ขากลับ: route เจาะจงชนะ default

    R2 หา route ที่ match 192.168.10.10 → มี match สองตัว: 0.0.0.0/0 (prefix ยาว 0 บิต) และ S 192.168.10.0/24 via 10.0.12.1 (prefix ยาว 24 บิต) → Longest Prefix Match เลือก /24 → forward ออก G0/1 ไป 10.0.12.1 — นี่คือเหตุผลที่ R2 ต้องมี route เจาะจงของ Branch LAN ไม่อย่างนั้นขากลับจะไหลออกไปทาง ISP ผิดทิศ

    SRC MAC: 0c2a.2b00.2001 (R2 G0/1) | DST MAC: 0c1a.2b00.1001 (R1 G0/1) | SRC IP: 8.8.8.8 | DST IP: 192.168.10.10
  6. R1 — ส่งเข้า LAN ปลายทาง

    R1 หา route ที่ match 192.168.10.10 → C 192.168.10.0/24, GigabitEthernet0/0 (connected /24 ชนะ default /0) → ปลายทางอยู่บน subnet ที่ต่อตรง R1 ทำ ARP หา MAC ของ 192.168.10.10 แล้วส่ง frame สุดท้ายเข้า LAN

    SRC MAC: 0c1a.2b00.1000 (R1 G0/0) | DST MAC: 0050.7966.6801 (PC1) | SRC IP: 8.8.8.8 | DST IP: 192.168.10.10
  7. PC1 — รับ Echo Reply

    PC1 ได้รับคำตอบ แสดงผล Reply from 8.8.8.8: bytes=32 time=2ms TTL=253 (TTL ขากลับเริ่มที่ 255 จาก IOS ถูกลดโดย R2 และ R1 รวม 2 ครั้ง) — ครบวงจรทั้งขาไปและขากลับ

    ผลบนหน้าจอ: Reply from 8.8.8.8: bytes=32 time=2ms TTL=253

Insight ที่ต้องจำขึ้นใจ

1) MAC address เปลี่ยนใหม่ทุก hop แต่ SRC/DST IP คงเดิมตลอดทาง (ไม่มี NAT ใน Lab นี้)

2) ทุก router ระหว่างทางต้องรู้จักทั้งขาไปและขากลับ — ping จะสำเร็จก็ต่อเมื่อ lookup ผ่านครบทุกจุดทั้งสองทิศทาง ขาดจุดเดียว = timed out เหมือนกันหมด ทำให้อาการภายนอกแยกไม่ออกว่าพังตรงไหน ต้องไล่ทีละ hop เท่านั้น

3) การ ping ที่ล้มเหลว บอกแค่ "วงจรไม่ครบ" ไม่ได้บอกว่าขาไหนขาด — traceroute ช่วยชี้จุดตายของขาไปได้ แต่ขากลับต้องเปิด routing table ปลายทางดูเอง

  • Design ฝั่งสาขา (R1): ทางออกเดียว → default route เส้นเดียวคือคำตอบที่ "ถูกทั้งทางเทคนิคและทางการดูแลรักษา" เพราะ HQ เพิ่ม subnet ใหม่กี่ครั้ง สาขาก็ไม่ต้องแก้ config
  • Design ฝั่ง HQ (R2): ต้องแยกทิศให้ถูก — ทิศสาขาใช้ route เจาะจง /24, ทิศ Internet ใช้ default — ตรงนี้คือหัวใจของ Longest Prefix Match: route เจาะจงกว่าจะดึงทราฟฟิกของสาขาออกจาก default โดยอัตโนมัติ
  • Design ฝั่ง ISP (R3): ISP ไม่ควรรู้จักเครือข่ายภายในของลูกค้ารายตัว จึงใช้ summary 192.168.0.0/16 เส้นเดียว — นี่คือภาพจำลองของหลัก aggregation ที่ทำให้ Internet ทั้งใบ scale ได้

เฉลยแบบฝึก LPM (Section 6):

  1. ปลายทาง 192.168.10.55 → match ได้ 2 route: 0.0.0.0/0 (0 บิต) กับ S 192.168.10.0/24 (24 บิต) → เลือก /24 via 10.0.12.1 เพราะ prefix ยาวกว่า — AD/metric ไม่ถูกนำมาคิดเลยเพราะ prefix ไม่เท่ากัน
  2. ปลายทาง 8.8.8.8 → ไม่มี route เจาะจงใดครอบ → match ได้เพียง S* 0.0.0.0/0 → ออก G0/2 ไป 10.0.23.2 — default route คือ "ตัวเลือกสุดท้าย" ที่ match ทุกอย่างแต่แพ้ทุกคน
  3. ปลายทาง 10.0.12.2 → match ได้ 3 route: 0.0.0.0/0, C 10.0.12.0/30 และ L 10.0.12.2/32 → เลือก L /32 ซึ่งคือ IP ของ R2 เอง — packet ไม่ถูก forward แต่ถูกส่งขึ้น process ภายใน router (นี่คือหน้าที่ของ local route ที่ IOS สร้างให้อัตโนมัติ)

สรุปกติกาสามชั้น: (1) LPM ตัดสินก่อนเสมอ (2) prefix เท่ากันจากหลาย source → AD ต่ำกว่าชนะ (3) source เดียวกัน prefix เดียวกันหลายเส้นทาง → metric ต่ำกว่าชนะ (ถ้า metric เท่ากันหลายเส้น จะเกิด load balancing)

Scenario 7.1 — R1 ping 8.8.8.8 ได้ แต่ PC1 ping ไม่ได้

สาเหตุจริง: R3 ไม่มี route กลับหา 192.168.10.0/24 — ขาไปสมบูรณ์ (R1 และ R2 มี default พาไปถึง 8.8.8.8) แต่ Echo Reply ที่ R3 สร้างมีปลายทาง 192.168.10.10 ซึ่ง R3 ไม่มี route จึง drop ทิ้งเงียบ ๆ ส่วนที่ R1 ping ได้เพราะ ping จาก router ใช้ source เป็น IP ขาออกคือ 10.0.12.1 ซึ่งอยู่ใน 10.0.12.0/30 ที่ R3 มี route (ตามใบร้องขอเดิม) พอดี — "ผลทดสอบสวยหรูที่หลอกเราเอง"

คำสั่งที่ใช้พิสูจน์ (บน R1 และ R3)

วิธีแก้: เพิ่ม ip route 192.168.0.0 255.255.0.0 10.0.23.1 บน R3 (หรือในโลกจริง: ส่งใบร้องขอ route เพิ่มไปที่ ISP) บทเรียน: ทดสอบด้วย source ของผู้ใช้จริงเสมอ — ping ... source คือเครื่องมือจำลองที่ประหยัดเวลาที่สุด และอย่าปิดงานด้วยการ ping จาก router เพียงอย่างเดียว

Scenario 7.2 — default route ใส่ next-hop ผิดเป็น 10.0.23.2 บน R1

สาเหตุจริง: next-hop 10.0.23.2 ไม่ได้อยู่ใน subnet ที่ R1 ต่อตรง (R1 ต่อกับ 10.0.12.0/30 เท่านั้น) การ resolve จึงต้องพึ่ง route อื่น แต่ route เดียวที่จะครอบ 10.0.23.2 ได้ก็คือ default route ตัวมันเอง — IOS ไม่ยอมให้ static route recursive ผ่านตัวเอง จึงไม่ install ลง RIB: บรรทัดอยู่ใน running-config ครบ แต่ routing table ไม่มี S* และ gateway of last resort "is not set"

การพิสูจน์บน R1

ip route 0.0.0.0 0.0.0.0 10.0.23.2
R1# show ip route | include Gateway
Gateway of last resort is not set
R1# show ip route 10.0.23.2
% Network not in table

วิธีแก้: no ip route 0.0.0.0 0.0.0.0 10.0.23.2 แล้วใส่ ip route 0.0.0.0 0.0.0.0 10.0.12.2 — next-hop ต้องเป็นเพื่อนบ้านบนลิงก์ที่ต่อตรงเสมอ (สำหรับ design ระดับนี้) บทเรียน: "running-config มี" ไม่เท่ากับ "route ใช้งานได้" ต้องตรวจ RIB ทุกครั้ง

Scenario 7.3 — พิมพ์ ip route ด้วย wildcard mask

สาเหตุจริง: สับสนระหว่าง subnet mask กับ wildcard mask — ip route ต้องการ subnet mask (255.255.255.0) แต่ผู้เรียนใส่ 0.0.0.255 ซึ่งเป็นรูป wildcard ที่ใช้กับ ACL / network statement ของ OSPF เท่านั้น IOS ตรวจว่า address AND mask แล้วไม่สอดคล้องกับ address ที่ให้มา จึงปฏิเสธด้วย % Inconsistent address and mask

คำสั่งผิด → คำสั่งถูก (บน R2)
R2(config)# ip route 192.168.10.0 0.0.0.255 10.0.12.1
% Inconsistent address and mask
R2(config)# ip route 192.168.10.0 255.255.255.0 10.0.12.1
! ตรวจผล: show ip route ต้องเห็น  S 192.168.10.0/24 [1/0] via 10.0.12.1

บทเรียน: จำเป็นคู่: ip route / interface = subnet mask, ACL / OSPF network = wildcard mask — และพึงระวังว่า wildcard บางค่า (เช่น 0.0.0.0) เป็น mask ที่ valid โดยบังเอิญ คำสั่งจะผ่านแต่ความหมายกลายเป็น host route ทันที ซึ่งอันตรายกว่า error เพราะไม่มีใครเตือน (เทียบกับ trap ใน initial config ของ Lab นี้)

แนวคำตอบ Reviewer Questions

Q1 — Longest Prefix Match ทำงานอย่างไร

เมื่อ packet มาถึง router จะนำ destination IP ไปเทียบกับทุก route ใน table โดยดูว่า "บิตต้น ๆ ของปลายทางตรงกับ prefix ของ route ครบตามความยาว prefix หรือไม่" route ทุกตัวที่ตรงถือว่า match แล้วเลือกตัวที่ prefix ยาวที่สุด ตัวอย่างจาก R2: ปลายทาง 192.168.10.55 match ทั้ง 0.0.0.0/0 (0 บิต) และ 192.168.10.0/24 (24 บิต) → ใช้ /24 ชี้ 10.0.12.1 — reviewer อยากได้ยินคำว่า "เจาะจงที่สุดชนะ และตัดสินก่อน AD/metric เสมอ"

Q2 — ทำไม 0.0.0.0/0 match ทุก IP

ความยาว prefix /0 หมายถึง "ต้องตรงกัน 0 บิตแรก" — ไม่มีเงื่อนไขให้ตรวจ IP ใดก็ผ่านเงื่อนไขโดยอัตโนมัติ ดังนั้น 0.0.0.0/0 จึง match ทุกปลายทางที่เป็นไปได้ แต่เพราะความยาว 0 บิตมันจึงแพ้ route อื่นทุกตัวที่ match เหมือนกัน — เป็น "ตาข่ายรองรับชั้นสุดท้าย" โดยธรรมชาติของ LPM ไม่ต้องมีกลไกพิเศษใด ๆ

Q3 — /24 vs /16 vs /0 (และถ้าคนละ protocol)

เลือก /24 เพราะ prefix ยาวที่สุด — จบแค่นั้น AD ไม่เกี่ยวเลยแม้ /24 จะมาจาก OSPF (AD 110) และ /16 มาจาก static (AD 1) ก็ตาม เพราะ AD ใช้เปรียบเทียบเฉพาะกรณี "prefix เดียวกันเป๊ะ ๆ จากหลาย source ว่าตัวไหนได้ลง table" ส่วนการ forward เทียบกันที่ความยาว prefix ของ route ที่อยู่ใน table แล้วเท่านั้น — คำถามนี้คือกับดักยอดนิยมในห้องสัมภาษณ์

Q4 — AD กับ metric ต่างกันอย่างไร

AD = ความน่าเชื่อถือของแหล่งที่มา (connected 0, static 1, eBGP 20, EIGRP 90, OSPF 110, RIP 120) ใช้ตัดสินเมื่อ prefix เดียวกันถูกเรียนรู้จากหลาย source — ตัวที่ AD ต่ำสุดได้ install ลง routing table metric = ความดีของเส้นทางภายใน protocol เดียวกัน (เช่น OSPF cost, RIP hop count) ใช้เลือก path ที่ดีที่สุดก่อนเสนอ route ให้ RIB ลำดับใช้งานจริง: LPM (ตอน forward) ← AD ← metric (สองตัวหลังเกิดตอนสร้าง table)

Q5 — ทำไม ISP ไม่ควรถือ route ละเอียดของลูกค้า

เหตุผลหลักคือ scale และภาระการเปลี่ยนแปลง: ISP มีลูกค้าหลักพันราย ถ้าถือ /24 รายเครือข่ายของทุกราย routing table จะบวมและทุกครั้งที่ลูกค้าเพิ่ม subnet ต้องเปิด ticket แก้ config ที่ ISP — เสี่ยงผิดพลาดและช้า แนวทางที่ถูกคือ summarization/aggregation: ตกลง block ที่ครอบทั้งหมดของลูกค้าแล้วถือ route เดียว เหมือน 192.168.0.0/16 → 10.0.23.1 บน R3 ใน Lab นี้ — ลูกค้าจัดสรร subnet ภายใน block ยังไงก็ไม่กระทบ ISP (บน Internet จริงหลักการเดียวกันนี้คือการ advertise aggregate ผ่าน BGP)

Q6 — ทำไม router ping ได้แต่เครื่องลูกข่าย ping ไม่ได้

เพราะ source IP ต่างกัน: ping จาก router โดย default ใช้ IP ของ interface ขาออก (ขา WAN) ซึ่งปลายทางมักมี route กลับ แต่เครื่องลูกข่ายใช้ IP ใน LAN ซึ่งปลายทางอาจไม่มี return route — พิสูจน์จากบน router ได้ด้วย ping <dest> source <LAN interface>: ถ้าแบบระบุ source ล้มเหลวขณะที่แบบธรรมดาผ่าน สรุปได้ทันทีว่าปลายทางขาด return route ไปยัง LAN subnet โดยไม่ต้องเดินไปที่เครื่องผู้ใช้เลย

Q7 — next-hop down แล้ว static route ยังอยู่ใน table ไหม

ต้องแยกกรณี: (1) interface ฝั่งเราที่ใช้ไปถึง next-hop down (ถูก shutdown / สายหลุด) → connected route ของ subnet นั้นหายจาก RIB → next-hop resolve ไม่ได้ → static route ถูกถอนออกจาก routing table (2) อุปกรณ์ปลายทางดับแต่ลิงก์ฝั่งเรายัง up (เช่น ต่อผ่าน switch กลาง) → connected route ยังอยู่ next-hop ยัง resolve ได้ → static route ยังค้างอยู่ทั้งที่ส่งจริงไม่ถึง กลายเป็น blackhole เงียบ ๆ — คำตอบระดับดีมากจะพ่วงว่า งานจริงแก้ด้วย IP SLA + track object ผูกกับ static route หรือเปลี่ยนไปใช้ routing protocol ที่มี keepalive/hello ตรวจเพื่อนบ้าน

จุดที่คนมักผิด

  • เห็น % Network not in table จาก show ip route 8.8.8.8 แล้วตกใจว่า route หาย ทั้งที่ default route ยังทำงานปกติ — คำสั่งนี้ไม่นับ gateway of last resort
  • ทดสอบด้วย ping จาก router แล้วปิดงาน โดยไม่เคยลอง source เป็นขา LAN หรือทดสอบจากเครื่องผู้ใช้จริง
  • ใช้ wildcard mask กับ ip route เพราะเพิ่งเรียน ACL/OSPF มา
  • copy ค่าจาก addressing table โดยไม่ตรวจทานกับ design (/30 ต้องเป็น 255.255.255.252 เสมอ)
